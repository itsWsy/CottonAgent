你是一名资深全栈工程师和 AI Agent 应用工程师。请在当前仓库中实现一个可运行、可演示、适合前端实习简历展示的 V1 项目。

# 一、项目概述

项目名称：棉智策（CottonPilot）
项目定位：面向棉花种植场景的农事推荐智能体平台。

用户可以创建棉田档案、录入历史农事操作、选择当前生育阶段、症状和天气标签，然后发起 Agent 分析任务。系统通过工作流依次读取棉田档案、读取历史操作、检索本地棉花领域知识、执行下一步农事推荐、生成 7 天计划，并通过 SSE 将执行步骤实时推送给前端。用户可以查看推荐依据并接受或拒绝方案。

该版本重点体现：
1. Vue3 + JavaScript 前端工程能力；
2. Agent 执行过程可视化；
3. SSE 实时任务状态；
4. Pinia 复杂状态管理；
5. 推荐结果的结构化展示；
6. FastAPI 与工作流编排；
7. 项目可以在没有大模型 API Key 的情况下完整运行。

# 二、重要开发原则

1. 不要只生成页面壳子、静态 Mock 页面或伪代码，必须实现可运行的前后端闭环。
2. 不要询问我确认，先检查当前仓库；如果仓库为空，直接创建完整项目。
3. 每完成一个阶段都运行必要的检查、测试或构建，发现错误立即修复。
4. 使用清晰、可维护的代码，不要把所有逻辑堆在单个文件。
5. 所有接口、数据结构、前端 JSDoc 定义和 SSE 事件保持一致。
6. V1 不实现图片识别、不训练深度学习模型、不接入向量数据库、不使用 Redis、不实现 MCP、不做多 Agent。
7. 不生成任何具体农药名称、剂量或自动执行建议。所有推荐只用于教学演示，并在界面明显位置提示“结果仅供辅助参考，实际农事操作需由专业人员确认”。
8. 不依赖大模型也必须运行。大模型调用是可选增强项，缺少配置或调用失败时必须自动使用模板生成最终说明。
9. 优先保证 Windows 环境可以启动，同时兼容 macOS/Linux。
10. UI 使用中文，代码变量使用清晰的英文命名。

# 十三、SSE 设计

前端使用 `@microsoft/fetch-event-source`，携带 Bearer Token 订阅：

GET `/api/agent/tasks/{task_id}/events`

不要使用无法携带自定义请求头的原生 EventSource。

SSE endpoint 连接后先发送 `task_snapshot`，内容包含当前任务、已存在步骤、推荐结果和计划，解决前端连接晚于任务开始、页面刷新或短暂断线时丢失事件的问题。

支持事件：

1. task_snapshot
2. step_start
3. step_success
4. step_failed
5. recommendations
6. farm_plan
7. evidences
8. answer
9. completed
10. failed
11. heartbeat

事件数据统一为 JSON，并至少包含：
- type
- taskId
- timestamp
- data

示例：
{
  "type": "step_start",
  "taskId": "xxx",
  "timestamp": 1782180000000,
  "data": {
    "stepId": "load_field_context",
    "stepName": "读取棉田档案"
  }
}

实现 `TaskEventBroker`：
- 单进程 V1 可使用 asyncio.Queue；
- 每个 taskId 维护订阅队列；
- 事件产生时先持久化数据库，再发布；
- 没有订阅者时不能影响任务执行；
- SSE 新连接必须通过数据库快照恢复当前状态；
- 每 15 秒发送 heartbeat；
- 任务完成或失败后发送最终事件并结束连接；
- 客户端断开时清理订阅队列。

候选操作来自：

3. 如果用户无历史记录，则使用当前生育阶段的全局高频操作。

基础转移得分：
transitionScore = 某操作在当前操作之后出现的次数 / 当前操作作为前序操作的总次数

生育阶段得分：
- 候选操作在当前生育阶段出现过：1
- 否则：0

知识匹配得分：
- 候选操作被 Top-3 知识推荐：1
- 否则：0

最终得分：
finalScore =
0.5 * transitionScore
+ 0.3 * growthStageScore
+ 0.2 * knowledgeMatchScore

要求：
- 得分保留两位小数；
- 按 finalScore 降序；
- 相同得分按 actionCode 稳定排序；
- 返回 Top-3；
- 每项包含 actionCode、actionName、score、expectedDay、reason、sourceType；
- 若候选不足 3 个，使用当前生育阶段高频操作补齐；
- 必须避免推荐与最近一次操作完全相同的无意义重复操作，除非知识明确要求“复查”。

推荐器单元测试至少覆盖：
- 有历史操作；
- 无历史操作；
- 候选不足；
- 稳定排序；
- 得分范围 0～1。

# 十七、7 天计划

建议天数映射示例：

每个计划项包含：

- actionCode
- actionName
- reason
- status：pending

计划按 day 升序，不能出现重复操作。
